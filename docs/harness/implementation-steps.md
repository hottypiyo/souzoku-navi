# 実装ステップ（ハイブリッドオーケストレーション）

AIによる実装（確率的）と自動検証（決定論的）を必ず交互に行うこと。

| ステップ | 種別 | 内容 |
|---|---|---|
| 1. 仕様確認 | 決定論 | `docs/specs/` または決定事項を読む |
| 2. AI実装 | 確率論 | コードを書く |
| 3. 静的検証 | 決定論 | `npx tsc --noEmit` + lint |
| 4. 動作確認 | 決定論 | `localhost:3000` で画面確認 |
| 5. 法令確認 | 決定論 | 数値・条文は e-Gov で一次確認 |
| 6. 記録・push | 決定論 | `docs/exec-plans/` に追記 → コミット → push |

**ステップ3〜5を省略したまま「完了」と報告しない。**

---

## 計画モードの発動基準（3ファイルルール）

以下のいずれかに該当する場合、実装前に計画を提示してから着手する：

- 変更するファイルが3つ以上
- 複数のディレクトリをまたぐ変更
- DBスキーマ変更（Supabase migration）を含む
- 外部API・法令データ連携を含む

計画提示 → 確認を得る → 実装。この順序を省略しない。

---

## 評価の視点

完了報告に「何をどう確認したか」を含める：

- TypeScriptエラー: ○件（0件が目標）
- 画面確認: 確認したフロー・ページ名
- 法令確認: 参照した条文・URL
