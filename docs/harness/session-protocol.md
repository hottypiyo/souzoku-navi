# セッションプロトコル

## セッション開始時

1. `git pull` で最新を取得
2. `docs/exec-plans/active/` の進行中プランを確認
3. `docs/errors.md` で既知のハマりポイントを確認
4. 作業対象の `docs/specs/` があれば読む
5. 法令・判例に関わる実装は `docs/legal/` を先に確認する

---

## セッション終了時のログ

コミット・push が完了したら、`docs/exec-plans/active/` の該当プランに以下を追記してから閉じる：

- 本日変更したファイルと内容
- 完了したこと・残っていること
- 次のセッションへの引き継ぎ事項

---

## エラーの学習化

実装中に遭遇したエラー・ハマりポイントは `docs/errors.md` に記録する。

形式: `YYYY-MM-DD | 変更箇所 | エラー内容 | 解決策`

---

## 黄金の原則（エントロピー管理）

1. 独自ヘルパーより共有ユーティリティを優先する
2. 法令数値・条文は推測で使わず e-Gov で一次確認する
3. ドキュメントが不十分なルールはコードに昇格させる
4. 劣化パターンはその週のうちに修正する
